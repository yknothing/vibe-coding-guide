# 把经典编译进环境：Agent Coding 时代的软件工程精华实践论

> 日期：2026-07-08
> 姊妹篇：[STRATEGY.md](./STRATEGY.md) 回答"这个仓库该变成什么"；本篇回答"变成它之后，拿什么内容去编译、按什么方法编译"。

---

## 摘要（十句话版本）

1. 经典软件工程书籍的作用机制是**改变作者写代码那一刻的行为**：读者读完，原则沉淀为直觉，直觉在指尖生效。
2. Agent 作为"读者"不可被改变——无梯度、无记忆、无后果；**能被改变的只有它所处的环境**。
3. 因此唯一可行的路径是**编译**：把书中的智慧从"给人的行为建议"翻译成"环境的构造"。
4. 这场编译的上半场早已发生：《Effective Java》被 Google 编译成了 Error Prone，C++ Core Guidelines 被编译成了 clang-tidy——但止步于机械章节，因为判断层章节只有人能执行。
5. LLM 第一次让**判断层可编译**：以"问题"形状存在的知识（红旗、评审提问）可以由批判者执行——极致，就是完成这场编译的下半场。
6. 统一定律是**验证不对称性**：Agent 的输出质量与验证信号的强度和廉价度成正比；编译的本质就是把主观品质转换为廉价的验证信号。
7. Agent 不是一个工程师，而是**所有工程师的叠加态**；上下文和代码库决定哪一个出场——范例与词汇是第二条编译通道。
8. Agent 的目标函数（完成本回合、让测试变绿）与经典最大的敌人——战术编程——**在结构上同构**；"软件工程是编程对时间的积分"，而 Agent 没有时间维度，时间必须由 harness 人工注入。
9. **极致不等于"和资深工程师一样好"，而是"执行人类团队从来负担不起的完整程序"**：每个设计比较过备选、变绿后必重构、复杂度只降不升、每个教训永久生效。
10. 资深性不住在模型里，住在仓库里——**仓库是整个系统中唯一会学习的组件**；人的角色收敛为四件事：提供易变性知识、策展持久工件与记忆、抽查品味、设计回路本身。

## 1. 经典到底说了什么：把书架压缩成六条原理

书很多，原理很少。《Clean Code》《A Philosophy of Software Design》《Effective Java/C++》《重构》《人月神话》《程序员修炼之道》《代码大全》《修改代码的艺术》《领域驱动设计》——把整个书架蒸馏到不可再压缩，剩下的大约是六条。每条都要回答同一个问题：**它在 Agent 时代发生了什么形变？**

### P1 复杂度是头号敌人，且以增量方式积累

> 出处：Brooks《没有银弹》（本质复杂度 vs 偶然复杂度）、Ousterhout（复杂度是增量的，来自千百次小妥协）、Moseley & Marks《Out of the Tar Pit》（可变状态是偶然复杂度之源）、Dijkstra《谦卑的程序员》（人的头骨有限）、Hoare 图灵奖演讲（"简单到明显没有缺陷，或复杂到没有明显的缺陷"）。

**形变**：代码的边际成本趋近于零之后，复杂度的增长速度乘以十——"写得多"从成就变成了风险敞口。稀缺资源从"写代码的时间"变成"理解代码的能力"，由此产生一个新债种：**理解债**——没有任何人（也没有任何一个上下文窗口）完整理解的代码在持续积累。战争还是那场战争，但双方的武器都工业化了：Agent 制造复杂度更快，而对抗复杂度的手段（棘轮、强制重构、廉价重写）也第一次变得工业化。

### P2 软件工程是编程对时间的积分

> 出处：Titus Winters《Software Engineering at Google》（"Software engineering is programming integrated over time"）、Lehman 软件演化定律、Fowler《重构》、Feathers《修改代码的艺术》。

一切工程纪律——测试、重构、文档、兼容性——本质上都是为"未来的变更"支付的保险。**形变**：Agent 恰恰没有时间维度。它失忆、不维护自己上个月写的代码、不为自己的设计决策买单。这是所有缺口中最深的一个：**时间维度必须由 harness 人工注入**——后果模拟、只升不降的质量棘轮、外化到仓库的记忆（见第 6 章）。

### P3 杠杆在边界：抽象与信息隐藏决定复杂度的分布

> 出处：Parnas 1972（按"什么可能变"来分解模块）、Ousterhout（深模块：简单接口藏住复杂实现）、GoF《设计模式》（边界形状的共享词汇表）、Evans《领域驱动设计》（限界上下文）、Hyrum 定律（可观察的行为终将被依赖）。

**形变**：边界获得了一个新的度量单位——**上下文预算**。深模块让 Agent 用少量 token 换取强大能力；泄露的抽象直接表现为 token 税和更高的出错率。模块的理想尺寸第一次有了可操作的定义：**一个任务加上它所依赖模块的接口，应当能装进一个舒适的上下文窗口**。信息隐藏从美学原则变成了性能优化。

### P4 反馈回路胜过先见之明

> 出处：Beck（TDD、极限编程）、Humble & Farley《持续交付》、Forsgren《Accelerate》（DORA 指标）、Gall 定律（"能工作的复杂系统无一例外从能工作的简单系统演化而来"）。

**形变**：Agent 是反馈回路的**放大器**。回路快而准，Agent 高速自我纠正；回路慢而糊，Agent 高速生产垃圾——放大的是回路的质量，不是代码的质量。因此**回路设计成为与模块设计同等重要的架构工作**：测试多快、报错多清晰、门禁多即时，直接决定 Agent 表现出的"资历"。

### P5 代码首先是写给人读的，其次才是给机器执行的

> 出处：SICP 序言（"程序必须写给人读，只是顺便让机器执行"）、《Clean Code》、Knuth 的文学编程。

**形变**：读者名单里加入了"带着有限窗口的 Agent"。可读性第一次变得**可度量**——理解一段代码所需的上下文大小。Clean code is context engineering。同时人的理解仍是最后防线：Simon Willison 的自律——"我不提交我无法向别人解释的代码"——是阻断理解债的最终门禁。

### P6 系统的结构映照生产它的组织的结构

> 出处：Conway 定律；《Team Topologies》把它反向利用（先设计团队边界，让架构随之成形）。

**形变**：「组织」现在包括 Agent 拓扑和任务切分方式。你按什么粒度切任务、怎么分配上下文窗口，代码就会长成那个形状——按文件切任务会得到文件级思维，按端到端特性切任务会得到贯通的设计。这既是风险也是杠杆：**上下文切分是新的组织架构决策**，可以做"逆 Conway 演习"的 Agent 版——把任务切分刻意设计成你想要的模块结构。

## 2. 为什么 Agent "懂"却"不做"：四个结构性缺口

这些书的全文和百万次讨论都在模型权重里。问题从来不是知识缺失，而是四个结构性缺口，使"知道"无法变成"做到"：

**G1 目标函数天生战术。** Ousterhout 整本书的靶子——战术程序员（tactical tornado，只求本次改动最快落地）——恰好就是 Agent 的默认目标函数："完成本回合任务、让测试变绿"。Agent 不是偶尔战术，是被目标函数注定战术。这是 P2 的直接推论：没有时间积分的主体，不可能自发做战略编程。

**G2 无后果、无记忆。** 资深工程师真正的老师不是书，是**维护自己烂决策的痛苦**。Agent 从不维护自己写的代码，每个会话都是新生——把知识压进直觉的那条反馈回路，对它整体断开。

**G3 单次采样、无比较。** "Design it twice"的洞察是：好设计不是第一个可行方案，而是备选方案比较后的幸存者。默认循环里的 Agent 对第一个可行设计采样一次就提交——它在做贪心搜索，而书要求锦标赛。

**G4 没有易变性知识。** 信息隐藏的决策依据是"什么可能变"（Parnas 的原始表述就是按变更隐藏决策）。这是产品知识和历史知识——在人的脑子里、在 git 历史里，不在任何默认喂给 Agent 的上下文里。不提供这个输入，它在原理上就做不对封装决策，写多少条"要封装"的规则都没用。

至于"为什么把原则写进 rule/prompt/skill 收效甚微"——注意力稀释、规则描述结果属性而非过程步骤、无强制回路——[STRATEGY.md](./STRATEGY.md) 第 2 章已有完整论证，此处不重复。结论一致：**问题不是知识注入不足，是作用机制错位。**

## 3. 编译论：改变不了读者，就改变环境

### 3.1 历史先例：编译的上半场早已发生

Google 把《Effective Java》中可机械判定的条目编译成了 **Error Prone**（编译期直接报错）；C++ 社区把 Core Guidelines 编译成了 **clang-tidy** 的 `cppcoreguidelines-*` 检查；《代码大全》的格式章节编译成了 formatter，直接从人类记忆里退役。

**上半场止步于语法层和机械层**，原因很简单：判断层的章节——模块该多深、什么该隐藏、这个抽象值不值得——只有人能执行。LLM 的出现移动了"可编译边界"：**以问题形状存在的知识，现在可以由一个带干净上下文的批判者执行**。极致，就是完成这场编译的下半场。

### 3.2 统一定律：验证不对称性

观察所有 Agent Coding 的成功与失败案例，规律只有一条：

> **Agent 的输出质量，与验证信号的强度和廉价程度成正比。**

有编译器、类型系统、快速测试这类廉价客观裁判的地方，Agent 表现接近资深；裁判昂贵而主观的地方（设计质量、命名品味、抽象深度），表现接近初级。三个推论：

1. Agent 时代工程建设的重心 = **为每一种你在乎的品质建造裁判**；
2. "编译经典"的严格定义 = 把书中的主观品质**转换为环境中的廉价验证信号**；
3. 自主性应当按验证覆盖度发放——验证强的任务类型可以整夜无人值守，验证弱的必须在设计节点上有人（见 6.5 的自主性阶梯）。

### 3.3 第二条通道：Agent 是叠加态，环境是采样器

模型见过 GitHub 上从顶级到糟糕的全部代码。Prompt 和上下文的作用不是"教会它"，而是**选择从哪个分布里采样**：你用"不变量、幂等、聚合根"的词汇说话，出场的就是说这种话的工程师；代码库里全是脚本味道的代码，出场的就是脚本作者。

这给了范例和词汇一个机制性解释：**代码库是最强的 few-shot prompt，统一语言是采样器配置**。策展好范例、清除坏范例、维护领域术语表——这些不是软性建议，是和门禁同等硬度的工程手段。

## 4. 编译手册：六种形态与经典对照表

每一条经典智慧都能编译为以下六种形态之一（判定顺序从上到下：能进门禁的绝不进流程，能进流程的绝不占常驻上下文）：

| 形态 | 定义 | 适用知识 |
|---|---|---|
| **门禁** | 确定性工具可判定，违规即报错 | 机械规则 |
| **流程** | harness 的硬步骤，不可跳过 | 过程性纪律 |
| **提问** | 批判者在干净上下文中必答的问题 | 判断性红旗 |
| **范例** | 仓库中策展的正反例与词汇 | 品味 |
| **经济学重估** | 人类因成本永远跳过、现在买得起的实践 | 被裁剪的建议 |
| **语义转译** | 含义在 Agent 时代发生变化、需重新表述的原则 | 需要新度量的 |

对照表（按经典出处组织，可持续扩充）：

| # | 经典出处 | 原始智慧 | 编译形态 | 落地机制 |
|---|---|---|---|---|
| 1 | Effective Java（机械条目） | equals/hashCode 契约、资源关闭等 | 门禁 | Error Prone / SpotBugs——历史先例，直接采用 |
| 2 | C++ Core Guidelines | 所有权、RAII、边界检查 | 门禁 | clang-tidy `cppcoreguidelines-*` |
| 3 | Clean Code | 命名达意、函数短小 | 门禁+提问 | linter 设尺寸上限；批判者审命名是否达意 |
| 4 | Fowler《重构》 | 两顶帽子：先让它工作，再让它正确 | 流程 | 变绿后的强制重构阶段——deadline 下第一个被人类裁剪的步骤，现在设为不可裁剪 |
| 5 | Beck《测试驱动开发》 | 红—绿—重构 | 流程 | 测试先行 = 给 Agent 一个先于代码存在的可验证目标；回合按红绿重构排序 |
| 6 | Ousterhout 红旗 | 浅模块、信息泄露、透传方法、时间分解 | 提问 | 干净上下文批判者的固定问题清单（红旗本来就是为"辨认"而写的） |
| 7 | Ousterhout "Design it twice" | 好设计是比较的幸存者 | 流程 | 计划期强制产出 2–3 个备选+权衡，单方案不予通过 |
| 8 | Parnas 信息隐藏 | 按"什么会变"分解模块 | 流程+提问 | 设计文档必答"隐藏了什么/什么易变"；变更压测（见 6.3） |
| 9 | Feathers《修改代码的艺术》 | 表征测试、接缝 | 流程 | hook 拦截：无表征测试不得修改遗留代码——先固定现有行为，再动手 |
| 10 | 《程序员修炼之道》曳光弹 | 先打通端到端，再充实 | 流程 | 最小可工作骨架先行，禁止大爆炸式生成（同为 Gall 定律的操作化） |
| 11 | 《程序员修炼之道》破窗理论 | 小的失修会加速整体腐化 | 门禁+范例 | 复杂度与坏味道棘轮；清除仓库中的坏范例与建立好范例同等重要 |
| 12 | Brooks《人月神话》 | 准备扔掉第一版（你反正会扔的） | 经济学重估 | 原型→带着学到的约束重写。人类时代太贵，现在一个回合的事 |
| 13 | Evans《领域驱动设计》 | 统一语言 | 范例+提问 | 领域术语表进项目记忆；批判者对照术语表审命名——同时也是采样器配置（见 3.3） |
| 14 | Chesterton 的篱笆 | 不明白用途的篱笆不许拆 | 提问 | 删除或重写前必须先解释这段代码为何存在（git 考古），答不出则停 |
| 15 | Knuth | 过早优化是万恶之源 | 流程 | 优化型改动必须附带测量证据（profile / benchmark），否则退回 |

三条值得展开：

- **第 6 条（红旗→提问）**藏着一个关键的不对称：**否定形知识远比肯定形知识可编译**。"模块要深"无法检验；"这个类的公开方法数接近它的实现行数"（浅模块症状）可以检验。把各书的正面原则翻译成它对应的"症状清单"，是编译判断层的核心手艺。
- **第 8 条（信息隐藏→变更压测）**是把美学变成实验：拿 2–3 个真实可能的下游需求分别砸向候选设计，让 Agent 实现它们，量 diff 的爆炸半径——变更局部化的设计胜出。用一个下午模拟六个月的维护，这是"design it twice"在时间维度上的推广，也是人类团队从来负担不起的实践。
- **第 9 条（表征测试 hook）**是"验证不对称性"的直接应用：遗留代码之所以危险，是因为它缺少裁判；那就先让 Agent 廉价地制造裁判（表征测试固定现状），再允许修改。Feathers 整本书由此压缩成一条 hook 规则。

## 5. 来自一线的印证：最好的 Agent Coding 实践都在做同一件事

以下实践出自不同的人和场景，但每一条都能映射回第 1 章的某条原理 × 第 4 章的某种形态——方向在收敛，因为约束（模型失忆、注意力有限、验证不对称）是同一组：

- **Anthropic《Claude Code 最佳实践》**：Explore → Plan → Code → Commit 工作流——"压力前移到设计时"的官方版本；并明确指出 TDD 在 agentic coding 中"变得更加强大"（P4）。
- **Anthropic《Building Effective Agents》**：evaluator–optimizer 模式——生成者/批判者分离的官方原型。
- **上下文工程共识**（Anthropic 的注意力预算论；Chroma 的 context rot 实证研究）：常驻规则按回合持续付注意力税——从机制上解释了"规则堆积"为何失效。
- **规格驱动开发**（GitHub Spec Kit、AWS Kiro）：规格是源码、代码是编译产物——"代码是牲畜、规格是宠物"的产品化。
- **干净上下文子代理 review**：不带生成时记忆的批判者已成为业界默认做法——防止让被告当法官。
- **Geoffrey Huntley 的 "Ralph" 整夜循环**：在验证覆盖强的仓库里，Agent 可以无人值守跑一整夜——"自主性 ∝ 验证强度"的极端实证。
- **Simon Willison**："我不提交我无法向别人解释的代码"——人类理解作为最终门禁，阻断理解债（P5）。
- **Armin Ronacher**：无聊的技术栈、快的测试、清晰的报错，Agent 表现显著更好——"Agent 是回路放大器"（P4）的一线证词。
- **Cognition《Don't Build Multi-Agents》**：上下文连续性重于并行度；每个动作都携带隐含决策，碎片化的对话交接必然互相冲突——角色分离必须靠持久工件（规格、设计文档）交接。
- **METR 2025 对照实验**：资深开源开发者使用 AI 后实际慢了 19%，而自我感觉快了 20%——不度量的"极致"是幻觉（见 6.5）。
- **Karpathy 的自主性滑块（autonomy slider）**：自主程度是逐任务调节的参数，不是开关。
- **Kent Beck**："我 90% 的技能一夜之间归零，剩下 10% 的杠杆放大了一千倍"——人的角色迁移最好的注脚（见第 8 章）。

## 6. 落地蓝图：五层作战地图

### 6.1 设计时（杠杆最大的一层，火力通常配得最少）

- 非平凡任务强制产出**设计文档**，必答四问：这个模块**隐藏**了什么（一句话答不出=什么都没隐藏）？哪些需求**易变**（人提供，这正是 G4 的解）？**否决**了哪些备选、为什么？**调用方**的代码长什么样（接口与用例先于实现）？
- 强制备选：2–3 个方案 + 权衡，单方案不予通过（表第 7 条）。
- **人 review 设计文档而不是 review 代码**——便宜一个数量级，且正好用在人的判断最不可替代的地方。

### 6.2 生成时

- **范例策展**：仓库即 few-shot；清除坏范例与建立好范例同等重要（3.3、表第 11/13 条）。
- **统一语言**：领域术语表进项目记忆，让"资深的那个工程师"出场。
- 项目记忆（CLAUDE.md 等）保持**瘦**：只放项目特有约定；通用知识在权重里，细节按需通过 skill 渐进加载。
- 门禁即时回注当前回合（机制见 STRATEGY.md 4.2）。

### 6.3 验证回路

- 测试先行：让可验证目标先于代码存在（表第 5 条）。
- 变绿后重构阶段**不可裁剪**（表第 4 条）。
- 复杂度与坏味道**棘轮**：基线只降不升，Agent 时代第一次可执行的"持续降复杂度"。
- **变更压测**：候选设计 × 可能的需求 → 量爆炸半径（表第 8 条）。
- 批判者：干净上下文 + 红旗问题清单 + 只判断不检测（检测归门禁）；同时专治门禁被应付（`const FOUR = 4`）。

### 6.4 记忆（仓库是唯一会学习的组件）

- ADR 与**悔棋记录**（"我们曾经 X，改成了 Y，因为 Z"）——书籍用 before/after 教学，这是对 Agent 最高带宽的品味传输。
- 定期 **git 考古**：变更聚集在哪、哪些模块放大变更，结论回写为项目记忆——git 历史就是被浪费的痛苦信号。
- 铁律：**人对 Agent 的每一次纠正，如果没有落成仓库里的某个工件（规则、范例、ADR、门禁），就只是一次性消费**。资深工程师把教训编译进直觉，Agent 系统必须把教训编译进仓库。

### 6.5 治理

- **自主性阶梯**：按任务类型的验证覆盖度发放自主权——覆盖强（有完整测试与门禁的重构、机械迁移）可整夜循环；覆盖弱（新领域设计）必须在设计节点上有人。
- **度量闭环**：返工率、变更放大系数（一个需求平均触碰多少文件）、time-to-green、review 发现密度、理解债抽查（随机抽一段代码，问团队谁能解释）。METR 的教训：感觉不可信，回路要用数据关闭。
- 人的四件事：提供易变性知识、策展持久工件与记忆、抽查品味、设计与调优回路本身。

## 7. 防坑：四个反模式

**AP1 规则堆积。** 已被实践证伪（本仓库的第一版就是标本）。每加一条常驻规则，都在向所有其他指令征收注意力税。处方：新知识入库前必须先回答"能否编译为门禁或提问"——都不能，才允许进常驻上下文，且要挤掉一条旧的。

**AP2 工件八股化。** 强制设计文档会退化成模板填空——Agent 极擅长生成看起来深思熟虑的八股。处方：问题少而锋利（四问足矣）；批判者交叉核对文档与代码的一致性；人抽查文档质量而非全查。

**AP3 过度脚手架。** 精巧的提示词编排会随下一代模型发布一夜作废——这是苦涩教训（Bitter Lesson）的 harness 版。处方：把投资押在"验证什么"（永恒）而非"怎么思考"（模型相关）。门禁、测试、规格、仓库记忆是**耐久层**；提示词技巧是**耗材**。

**AP4 上下文碎片化。** 用多 Agent 并行来追求吞吐，把决策切碎在互相看不见的对话里，每个动作携带的隐含决策互相冲突（Cognition 的核心教训）。处方：角色可以分离（设计者/实现者/批判者），但交接必须通过持久工件——规格、设计文档、ADR——而不是对话转述。

## 8. 结语：极致的重新定义

Agent 时代实践软件工程精华，本质上是完成三个转移：

1. **知识的居所**：从人脑 → 环境（门禁、流程、提问、范例）；
2. **纪律的执行**：从自觉 → 回路（自觉在 deadline 前必然让步，回路不会）；
3. **经验的积累**：从个人直觉 → 仓库工件（模型不学习，仓库学习）。

"极致"由此重新定义。它不是让 Agent 模仿最好的人类团队——那个标杆定低了。它是执行人类从来负担不起的完整程序：**每个设计都比较过备选，每行遗留代码修改前都有表征测试，每次变绿后都重构，复杂度每次触碰只降不升，每个教训都永久生效**。这套程序的每一条都出自几十年前就写下的书；新的不是思想，而是思想第一次变得可以被完整执行。

Kent Beck 说他 90% 的技能归零、剩下 10% 的杠杆放大了一千倍。那 10% 恰好是书教不会、也编译不掉的部分：判断什么值得做、什么会变、什么是好品味——以及设计让其余 90% 自动发生的那个回路。

这也是本仓库的使命：从"给 AI 读的知识库"，变成"把经典编译进环境"的工具链与方法论。
