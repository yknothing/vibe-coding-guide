# 编译《Effective Java》：一个已经发生的先例研究

> 选它的原因与 APoSD 相反：APoSD 展示编译的**前沿**（判断层如何编译），本书展示编译的**历史**——90 个条目在过去十五年里被三波力量逐层吸收，是"编译阶梯"（[README.md 第 1 节](./README.md)）真实运转过的完整标本。研究它回答一个问题：**一条智慧被编译之后会发生什么，以及还剩下什么。**

---

## 1. 三波编译：一本书如何消散于语言与工具之中

### 第一波：语言吸收（L1——最高层级）

| 条目 | 原始形态 | 吸收方式 |
|---|---|---|
| Item 9：try-with-resources 优于 try-finally | 手写纪律 | Java 7 语言特性，资源关闭成为语法 |
| Item 10/11/12：覆盖 equals/hashCode/toString 的契约 | 三章手写规则，历史上错误率极高 | Java 16 records 直接生成正确实现——**三个条目整体退役** |
| Item 42–44：优先 lambda 与方法引用 | 匿名类样板 | Java 8 语言特性 |
| Item 55：审慎返回 Optional | null 约定 | 类型系统承载"可能缺席"语义 |
| Item 34–38：用枚举替代 int 常量 | 命名纪律 | Java 5 枚举吸收枚举相关纪律；Java 17 sealed 是现代 Java 对受限类型集的补充机制 |

**观察**：语言吸收是终极编译——被吸收的条目不再需要任何人（或 Agent）"记得"，违反变得不可表达。这就是"技术栈选择 = 预编译份额选择"的实证：**选 Java 21 而不是 Java 8，会让多个 Effective Java 条目的机械部分由语言或标准库直接承担**。

### 第二波：工具吸收（L2–L3）

| 条目 | 吸收工具 |
|---|---|
| Item 40：坚持使用 @Override | Error Prone `MissingOverride`（默认 `WARNING`，可由构建配置升级为失败） |
| Item 11：equals/hashCode 成对契约（存量代码） | Error Prone `EqualsHashCode` |
| null 纪律（Item 54/55 周边） | NullAway / Checker Framework——把 null 安全提升到类型层 |
| Item 17：最小化可变性 | Immutables/records + Error Prone `Immutable` 对标注为 `@Immutable` 的类型做深不可变性检查 |
| Item 26–29：泛型与原生类型 | 编译器警告 + `-Werror` 门禁 |

**观察**：Error Prone、NullAway、Checker Framework 等工具覆盖或呼应了若干 Effective Java 机械条目，是经典规则工具化的历史先例；但它们不是整本书的完整实现，也不能替代 API 设计品味。这正是本登记册要对判断层继续推进的事。

### 第三波：待编译的判断层（L5–L6——本登记册的工作面）

剩下的条目全部是"何时/是否/值不值"形态，机械工具够不着，恰好落在 LLM 批判者的新边界上：

#### EJ_18 组合优于继承

- **原始智慧**：跨包继承具体类是脆弱的——子类依赖父类的实现细节，父类演化即破坏子类（Item 18）。
- **编译**：L6 批判者——每个新增的 `extends` 必答："这里是 is-a 还是想复用代码？父类为继承做过设计（受保护钩子、文档化自用模式）吗？换成组合+转发的代价是什么？"
- **Goodhart 申报**：为绕过审查改用"接口 + 复制实现"，制造重复。对策：DRY 门禁（v1 FND_002）反向制衡。
- **状态**：T

#### EJ_19 要么为继承而设计并写文档，要么禁止继承

- **原始智慧**：可继承是一份沉重契约（必须文档化自用模式），不打算履约就 final（Item 19）。
- **编译**：**L3 可下推**——"final by default" lint：公开非 final 类若无 `@DesignedForInheritance` 类注释（附自用模式文档）即报错。判断层条目中少数能整体下推的，优先实现。
- **Goodhart 申报**：无脑给所有类加注释。对策：注释要求附带文档段落，批判者抽查文档实质性。
- **状态**：T

#### EJ_1/2 静态工厂与 Builder

- **原始智慧**：构造器多参数即失控；静态工厂有名字、可缓存、可返回子类型（Item 1、2）。
- **编译**：L3（构造器参数数阈值，v1 DSN_004 已有）+ L7 范例——仓库内建立一个标准 Builder 范例，Agent 自然模仿。
- **状态**：I（机械检测指向生态现成 lint，范例策展归 L7。注意：v1 DSN_004 从未接线，不构成已覆盖的依据——那正是把图纸当门禁）

#### EJ_64 通过接口引用对象

- **编译**：L3——现成 lint 可查声明类型；例外（需要具体类方法）由批判者裁决。
- **状态**：T

#### EJ_57–62 局部变量作用域、for-each、精确的数值类型

- **编译**：L3——全部现成 lint。列出仅为完整性；**不进常驻上下文，不进批判者清单**（上推原则：够得着 L3 的绝不占 L6）。
- **状态**：I（指向现成工具；本仓库接线后按条目转 T→P）

## 2. 这个标本教给登记册的四件事

1. **编译是单向棘轮，而且是好事。** 每个条目的宿命是向上迁移（L6 → L3 → L1）直到消失。judged-by-critic 只是条目生命周期的中间站，不是终点。登记册的生命周期规则见 [README 第 5 节](./README.md#5-反身性harness-自身也是软件)。
2. **书的层级混杂，必须拆开分层。** 《Effective Java》一本书里同时有 L1 候选（已被 records 吸收）、L3 候选（@Override）和 L6 候选（组合优于继承）。**按书组织阅读，按层组织执行**——这是登记册与读书笔记的本质区别。
3. **升级技术栈是最便宜的编译动作。** 在为 Agent 建设门禁之前，先把语言版本、strict 模式、已有静态分析器拉满——那是别人替你编译好的几十个条目，一行配置的价格。
4. **残渣与分类学吻合——注意这是定义框架的自洽，不是实证验证**（残渣本就被定义为编译后剩下的部分）。全书唯一无法编译的部分是隐含在字里行间的"API 设计品味"（什么该进公共 API、什么概念值得命名）——正是 README 第 4 节的第 1、3 类残渣。书能明说的都能编译；只能示范、不能明说的部分，本来就靠 Bloch 的品味，现在靠你的。
