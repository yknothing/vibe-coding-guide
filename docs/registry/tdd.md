# 编译《测试驱动开发》（Test-Driven Development: By Example, Beck）

> 选它的原因：TDD 是"验证不对称性"最古老的工程化实践——在代码存在之前先制造裁判。Anthropic 的官方最佳实践明确指出 TDD 在 agentic coding 中"变得更加强大"；本文回答的是**为什么更强大、强在哪一步、以及 Agent 会怎么应付它**。
> 条目格式见 [README.md](./README.md) 第 2 节。

---

## BCK_01 红灯必须被实际观察到（核心条目）

- **原始智慧**：红→绿→重构。先写一个**失败的**测试，看它失败，再写实现让它通过（第一部分全篇节奏）。
- **Agent 时代的转译**：人做 TDD 时"看到红灯"是自然发生的；Agent 惯于**测试与实现一把生成**，测试从未证明过自己有牙齿——事后测试往往只是把实现（连同它的 bug）抄写成断言。"红灯实证"是整个 TDD 中对 Agent 最关键、也最容易被静默跳过的一步。
- **编译**：L5 硬步骤——测试写完必须由 **harness 执行并记录**，不采信 Agent 转述。红灯记录至少包含测试命令、失败输出、时间戳、工作区树 hash、测试文件 diff、实现文件是否已存在或已变更。允许两段式红灯：编译/导入失败只能算 provisional red；补齐最小 stub 后必须观察到与预期行为相关的断言红灯，才允许进入实现。若红灯快照前实现已存在或实现文件已被改动，标记为 `post-hoc test`，不满足本条。
- **验证信号**：变异测试杀死率 / 存活率是主判据；红灯实证执行率只是第一道滤网；逃逸缺陷中"测试本可捕获但断言空洞"的占比用于周期校准。
- **Goodhart 申报**：四种应付——(a) 先写实现、藏起或暂存，再补红灯制造创作顺序假象；(b) 写一个无关必败断言充当红灯；(c) 把编译/环境错误冒充有效断言红灯；(d) 红灯后写实现时顺手弱化断言。对策：红灯前后工作区快照由 harness 记录；红→绿之间测试文件 diff 受审，断言只许增强不许减弱；provisional red 必须升级为断言红灯；关键路径或抽样运行变异测试，存活 mutant 必须修测试或登记豁免。
- **状态**：T —— **本书优先级最高的条目**

## BCK_02 测试清单先行

- **原始智慧**：动手前把想到的所有测试场景写成清单（test list），逐项消灭；清单是安全感的来源，也是范围的边界（第 1 部分）。
- **编译**：L5 流程——测试清单作为计划工件，是[设计文档五问](./README.md#设计文档五问流程类条目的共享工件)向实现层的自然延伸：**规格的最细粒度形态就是一张测试清单**。实现过程逐项点亮，清单外的行为变化即范围蔓延信号。
- **验证信号**：清单覆盖的场景与逃逸缺陷的对照（逃逸缺陷是否属于"清单本该列出"的类别）。
- **Goodhart 申报**：清单只列 happy path。对策：清单模板强制按维度自查——边界值、异常路径、并发/重入、幂等——每个维度要么有条目，要么显式写"不适用及理由"。
- **状态**：T

## BCK_03 三角化：用第二个例子逼出泛化

- **原始智慧**：一个测试例子不足以约束实现——"fake it"（硬编码返回期望值）就能通过；增加第二个不同的例子，才**迫使**实现泛化（第 1 部分，triangulation）。
- **Agent 时代的转译**：这条对 Agent 的意义远超原书。LLM 本质上是从欠约束规格自由发挥的机器——单例子规格下，它生成的"泛化"是从训练分布里猜的，不是从你的意图里来的。**三角化是给生成器上约束的系统方法**：例子加到实现被唯一确定（或确定到你在乎的精度）为止。讽刺的是，Beck 描述的"fake it 然后被第二个例子逼真"的作弊者，行为模式与 Agent 完全同构——本书 1994 年就写好了 Agent 的行为预测。
- **编译**：L5/L6——关键行为的测试至少两个相异例子（L3 可数）；批判者升级问题："现有测试集合下，一个恶意的最小实现能骗过它们吗？"——这就是变异测试的思想实验版，昂贵场景下再跑真的变异测试（L4）。
- **验证信号**：变异测试存活率（存活的变异体 = 欠约束的证据）。
- **Goodhart 申报**：机械凑第二个例子（与第一个同构，约束力为零）。对策：批判者审例子间的**判别力**——两个例子若被同一个硬编码骗过，不算三角化。
- **状态**：T

## BCK_04 难测即设计信号（跨书元模式）

- **原始智慧**：测试难写不是测试问题，是被测对象耦合过高、依赖不清的设计问题（贯穿全书；《Growing Object-Oriented Software, Guided by Tests》一书将其发展为完整方法论）。
- **编译**：L6 批判者——Agent 报告"这个测试需要 mock 五个依赖才能搭起来"时，正确响应不是硬写 mock，而是回到设计。
- **元模式**：至此登记册出现了第三个同构条目——**写不出名字**（APOSD_14）、**写不清注释**（APOSD_12）、**搭不起测试**（本条）。三者是同一个信号的三个探头：**表达困难即设计缺陷**。批判者遇到任何一个探头报警，标准动作都是建议拆分/解耦，而不是在表达层硬憋。这个元模式本身值得进入批判者的顶层指令。
- **Goodhart 申报**：为"好测"而过度注入依赖，制造穿透所有层的构造参数瀑布。对策：与 [APOSD_08 复杂度下沉](./aposd.md#aposd_08-把复杂度往下拉) 制衡——可测性不得以调用方复杂度为代价，冲突时升级人裁。
- **状态**：T

## BCK_05 TDD 的真实产品是执行性规格（索引条目）

- **原始智慧**：TDD 的产出不只是"有测试的代码"，而是一份**机器可执行的需求陈述**。
- **编译**：语义转译已在姊妹篇完成（代码是牲畜、规格是宠物；测试是需求的机器可读形态）。代码可再生的时代这条被放大：实现丢了可以从测试重建，测试丢了实现就成了无主行为。本条仅作索引。
- **状态**：I（索引）

## 实现优先级建议

| 顺序 | 条目 | 理由 |
|---|---|---|
| 1 | BCK_01 红灯实证 | 一条 hook 规则封堵"测试自证无牙"这个最大漏洞，成本极低 |
| 2 | BCK_02 测试清单 | 与设计文档流程天然衔接，纯流程改动 |
| 3 | BCK_03 三角化 + 变异抽查 | 依赖变异测试基础设施，排在仪表盘（APOSD_02a）之后 |
| 4 | BCK_04 元模式进批判者顶层指令 | 一句话的改动，但需要批判者框架先就位 |
