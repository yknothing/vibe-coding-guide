# 编译《修改代码的艺术》（Working Effectively with Legacy Code, Feathers）

> 选它的原因：这本书对 Agent 时代有特殊地位——Agent 的最高危行为不是写新代码，而是**修改没有测试的旧代码**（"顺手重构"一段无人理解、无验证覆盖的逻辑）。Feathers 整本书恰好是这类操作的手术规程，且压缩率惊人：核心可以编译成一条 hook、一个默认策略和一本按需加载的手册。

---

## 1. 核心重定义：遗留代码 = 没有测试的代码

- **原始智慧**：遗留代码与年龄无关、与作者无关——**没有测试的代码就是遗留代码**（前言）。这个定义的价值在于它是操作性的：不需要判断，只需要查覆盖。
- **编译**：L3 触发器——这不是一条规则，而是**整个手术规程的开关**：diff 触碰的代码若无测试覆盖，以下所有条目自动激活。昨天 Agent 自己生成的无测试代码，同样按遗留代码处置（这个推论在 Agent 时代比原书更重要：**Agent 高速生产着"出生即遗留"的代码**，除非测试与代码同回合出生）。
- **Goodhart 申报**：生成断言空洞的伪测试骗过覆盖率。对策：变异测试抽查（改动实现，伪测试不红即曝光）；批判者审断言实质性。
- **状态**：T

## 2. FTH_01 表征测试先行（核心条目）

- **原始智慧**：修改遗留代码前，先写**表征测试**（characterization tests）——不验证"正确行为"（没人知道什么是正确），只**固定现有行为**：喂输入、记输出、断言现状（第 13 章）。
- **Agent 时代的转译**：表征测试对人是苦役（大量机械的输入枚举），**对 Agent 是最擅长的工种**——成本结构反转使"先表征后修改"第一次可以成为硬性规程而非奢侈品。这是"验证不对称性"的直接应用：遗留代码危险是因为没有裁判，那就先让 Agent 廉价地**制造裁判**，再允许修改。
- **编译**：L5 hook——`触碰无覆盖代码 → harness 计算本回合将修改的行 → 先生成覆盖这些行的表征测试并确认全绿 → 才允许修改 → 修改后表征测试红灯按分级确认`。覆盖工具不可用时，替代路径是由不知道修改意图的独立上下文生成输入集合，再由 harness 记录输入、输出和审批结果。
- **验证信号**：修改行覆盖映射；表征测试捕获的意外行为变更次数；变异得分 / 变异抽查存活率；遗留代码区的缺陷逃逸率。
- **Goodhart 申报**：三种应付——(a) 只表征不打算改的路径；(b) 用空断言或快照噪声刷覆盖；(c) 全有全无触发过重，用户直接关闭 hook。对策：表征测试必须覆盖本回合将修改的行；覆盖率只算入口，不算质量证明；关键路径或抽样运行变异测试，存活 mutant 必须修测试或登记豁免；红灯确认分级，少量逐条确认，大批量按模块梯度审批；默认边界值和异常路径各不少于 3 例（`cfg: FTH01_MIN_EDGE_CASES=3`），项目可调但必须记录理由。
- **状态**：T —— **本书优先级最高的条目**

## 3. FTH_02 Sprout 优先：在烂代码上加功能的默认姿势

- **原始智慧**：往遗留代码里加功能有两种安全姿势——**Sprout**（新逻辑写成新方法/新类，在旧代码里只加一行调用）和 **Wrap**（包住旧入口，在包装层加新行为）。共同原则：**新代码生而有测试，旧代码改动最小化**（第 6 章）。
- **Agent 时代的意义**：这是对 Agent"顺手重构"倾向的直接约束。Agent 读到烂代码会倾向于当场重写（它没有恐惧感），而无表征测试的重写是最高危操作。Sprout 把风险拓扑反转：**增量全部落在可验证的新代码里**。
- **编译**：
  - L7 范例 + 项目记忆一条短规则："修改无覆盖代码时默认 Sprout/Wrap，重写需先满足 FTH_01"；
  - L3 门禁：无覆盖文件的 diff 行数预算（例如单回合 ≤ N 行），超预算强制走表征测试流程。
- **Goodhart 申报**：Sprout 滥用——所有逻辑都长在"新芽"里，旧代码成为僵尸躯干，系统碎成两个世代。对策：git 考古监测"僵尸模块"（只被绕开、不再被理解的旧核心），达到阈值触发正式的表征+重写任务。
- **状态**：T

## 4. FTH_03 接缝与解依赖目录 → 按需加载的手术手册

- **原始智慧**：**接缝（seam）**= 不改代码就能改变行为的位置；书的后半部是 24 种解依赖技术目录（参数化构造器、提取接口、包装静态调用……），用于把测不了的代码变成测得了的（第 4、25 章）。
- **编译**：L6 技能（skill）——这是"目录形知识"的标准编译形态：**不进常驻上下文，做成按需加载的手术手册**。触发条件明确（FTH_01 激活但依赖太硬写不了测试时加载），内容就是解依赖技术的决策树 + 每种技术的最小示例。
- **Goodhart 申报**：无。此条目是能力供给而非约束，无应付面。
- **状态**：T

## 5. FTH_04 草稿重构：为理解而重写，然后扔掉

- **原始智慧**：看不懂的代码，动手重构一遍**只为理解**，然后 revert——理解留下，改动扔掉（Scratch Refactoring，第 16 章）。原书中这是奢侈品（人力太贵）。
- **编译**：经济学重估——对 Agent 这是**零成本读法**：在隔离工作区重写目标模块，产出物不是代码而是**理解**（写成模块级文档/注释回填），diff 丢弃。可作为 FTH_01 的前置步骤：先草稿重构以理解行为，再写表征测试。
- **Goodhart 申报**：草稿产出的"理解文档"八股化。对策：与 APOSD_12 的盲测判据共用——新会话 Agent 仅凭该文档能否正确预测模块行为。
- **状态**：E（实验性——值得一试的新实践，人类时代不存在对应物）

## 6. 一处需要反向修正的地方

Feathers 的整体姿态是**微创**：最小化改动、尊重现状——因为 2004 年改动昂贵、验证昂贵。表征测试把验证变廉价之后，这个前提被部分推翻：**有表征测试护航时，整模块重写常常优于长期微创**（微创累积的是 Sprout 疤痕组织）。修正后的决策规则：

- 无表征测试 → 微创（Sprout/Wrap），Feathers 全套照旧；
- 表征测试就位 → 重写进入选项，用 [APoSD 三类复杂度信号](./aposd.md#aposd_02a-复杂度的定义与仪表盘)对比微创与重写两个候选的复杂度前景，择优。

这是登记册"语义转译"形态的又一例：书没有过时，但它的成本假设过时了，结论需要带着新价格表重算。

## 实现优先级建议

| 顺序 | 条目 | 理由 |
|---|---|---|
| 1 | FTH_01 表征测试 hook | 直接封堵 Agent 最高危操作路径 |
| 2 | FTH_02 Sprout 默认策略 | 一条项目记忆规则 + 一个行数预算门禁，成本极低 |
| 3 | FTH_03 解依赖技能 | 依赖 FTH_01 落地后才有触发场景 |
| 4 | FTH_04 草稿重构 | 实验性，供探索 |
