# 编译《A Philosophy of Software Design》（Ousterhout）

> 选它做第一个完整样本的原因：这是正典中**判断层知识密度最高**的一本——几乎没有可以直接进 linter 的机械规则，全书都是"何时、多少、值不值"的权衡。如果编译方法论对它成立，对其他书只会更容易。
> 条目格式见 [README.md](./README.md) 第 2 节。所有条目初始状态为 T（试验中），实现并通过度量验证后转 P。
> 章节号按 Second Edition（2021）标注；第一版读者需注意第二版新增第 21 章，并重写/扩展第 6 章。

---

## 第一部分：复杂度与战略（第 1–3 章）

### APOSD_02a 复杂度的定义与仪表盘

- **原始智慧**：复杂度不是"代码难不难看"，而是三个可观测症状：变更放大、认知负荷、未知的未知（第 2 章）。
- **编译**：L3/L4 度量 —— 三症状对应三类信号（变更放大系数 / 上下文足迹 / 缺陷逃逸相关性），已提升为整个登记册的地基，详见 [README.md 第 3 节](./README.md)。
- **采集协议**：最小原型为 `python3 tools/complexity_dashboard.py --since <rev> --format json`；由 release/PR harness 提供 Git 范围，由 agent harness 可选提供 context JSON/JSONL，由缺陷回溯流程可选提供 defect JSON/JSONL。缺失数据必须显示为 absent，未知缺陷来源必须显示为 unknown。
- **验证信号**：这些信号与返工率、review 发现密度、逃逸缺陷的预测性校准；单个趋势不得独立证明复杂度下降。
- **Goodhart 申报**：为降低"每文件"指标而把大文件拆成高耦合的小文件碎片；为降低上下文足迹而少读多猜；为降低缺陷逃逸相关性而延迟或漏报缺陷。对策：变更放大按**模块和逻辑变更**统计；上下文足迹只和缺陷/返工联合判读；缺陷来源清单固定，unknown 来源单独计入。
- **状态**：T

### APOSD_02b 复杂度的两个根源：依赖与晦涩

- **原始智慧**：一切复杂度来自依赖（改这里必须懂那里）和晦涩（重要信息不在场）。
- **症状**：晦涩的操作性判据——读懂这段代码需要哪些**不在文件里**的知识？
- **编译**：依赖 → L3：import/耦合图度量与门禁。晦涩 → L6 批判者必答题："列出理解此 diff 所需但不在 diff 及其直接依赖中的背景知识；每一条都是一处晦涩，要么内联为注释/命名，要么说明为什么不。"
- **验证信号**：批判者列出的"场外知识"条数随时间下降。
- **Goodhart 申报**：用大段冗余注释稀释问题而非改善命名与结构。对策：注释重复代码本身是红旗（见 APOSD_12），两个批判者问题互为制衡。
- **状态**：T

### APOSD_03 战略编程 vs 战术编程

- **原始智慧**：战术编程（每次只求最快落地）是复杂度的主要来源；战略编程要求持续投资约 10–20% 在"让未来更容易"上。**Agent 的默认目标函数与战术编程同构，这条是全书对 Agent 最重要的一条。**
- **编译**：L5 流程 —— 战略投资不能靠嘱咐，只能靠回合结构强制："测试变绿"之后必须有"重构与清理"阶段才允许结束回合（Fowler 两顶帽子的执行化）；配合回合级棘轮（触碰过的代码，认知复杂度、坏味道计数、修改行覆盖率等回合内可计算量不准劣化）。
- **采集协议**：最小棘轮已接入 `hooks/post_tool_use_quality_gate.py --ratchet-baseline <previous-report.json>`；baseline 来自该 hook 的 JSON 输出，当前覆盖触碰文件的魔法值计数、硬编码端点计数和最大 Python 函数复杂度。强制"绿灯后重构阶段"仍待上层 harness 接线。
- **验证信号**：回合级棘轮曲线；变更放大系数、上下文足迹、缺陷逃逸相关性的周期趋势。
- **Goodhart 申报**：表演性重构——挪动代码、改名、重排，实际复杂度不变，用来满足"重构阶段必须有产出"的隐含预期。对策：重构阶段**允许空产出**（"检查后无需重构"是合法结论），但若有产出，必须伴随至少一项度量的实际改善；批判者抽查重构 diff 的实质性。
- **状态**：T

## 第二部分：模块设计与错误语义（第 4–11 章）

### APOSD_04 模块应当是深的

- **原始智慧**：模块的价值 = 功能强度 − 接口复杂度。好模块用简单接口藏住强大实现；浅模块（接口和实现一样复杂）是负资产（第 4 章）。
- **症状**：浅模块红旗——公开方法数、参数总数与实现行数同一量级；调用一次它，和自己写一遍差不多费劲。
- **编译**：
  - L3 信号（非硬门禁）：**接口表面积/实现体积比**（公开方法数 + 公开参数总数 + 公开类型数，除以实现行数），异常值提示批判者关注；
  - L6 批判者必答："这个模块替它的使用者**消化**了什么复杂度？"答不出即浅模块；
  - 语义转译：深模块 = 低上下文足迹——Agent 用这个模块完成任务需要读多少 token，是深度的代理信号。
- **验证信号**：模块的上下文足迹；调用方代码的行数变化。
- **Goodhart 申报**：把多个方法合并成一个巨型方法、把参数打包成一个万能 options 对象来压低"表面积"。对策：认知复杂度门禁管住方法体；options 对象的字段数计入表面积。
- **状态**：T

### APOSD_05 信息隐藏与信息泄露

- **原始智慧**：模块该隐藏的是**设计决策**（Parnas 的原始表述），尤其是可能变化的决策；泄露 = 同一个决策的知识出现在多个模块里（第 5 章）。
- **症状**：时间分解红旗——按"先做什么后做什么"而不是按"谁知道什么"切模块（`readFile / processFile / writeFile` 三个类共享文件格式知识）。
- **编译**：
  - L5 流程：[设计文档五问](./README.md#设计文档五问流程类条目的共享工件)第 1、2 问——"这个模块隐藏了什么决策？哪些需求易变？"（易变性由人注入，这是残渣清单第 2 条的注入接口）；
  - L4 变更压测：由不知道本命方案的独立上下文，从大于且不等于已披露易变性清单的场景池里抽取 2–3 个真实可能的需求变更，分别施加到候选设计；爆炸半径按"触碰模块数 + 接口签名变更数"记录——**信息隐藏质量的实验判据**，取代美学争论；
  - L6 批判者："找出在两个以上模块中出现的同一项知识（格式、协议、约定、魔法值）。"
- **验证信号**：变更压测的爆炸半径；变更放大系数。
- **Goodhart 申报**：两种应付——(a) 探针实现者即被评方，故意把 diff 做小；(b) 压测场景与设计时披露的易变性答案同源，等于泄题。对策：探针变更由独立上下文实现；场景池由人维护并大于当轮披露清单，抽样记录入库，场景池定期轮换。
- **状态**：T

### APOSD_06 略微通用的接口更深

- **原始智慧**：接口按"这类问题"设计而非按"今天这个调用方"设计，通常更简单也更深；但实现仍然只做今天需要的（somewhat general-purpose：接口通用、实现特化）（第 6 章）。这与 YAGNI 不冲突——YAGNI 管功能范围，本条管接口形状。
- **症状**：special-general mixture 红旗——通用机制里嵌着某个调用方的特例知识。
- **编译**：L6 批判者两问："把当前唯一的调用方遮住，这个接口还能自述其含义吗？""接口文档里是否出现了某个具体调用方的业务词汇？"
- **验证信号**：第二个调用方出现时接口是否无需修改（事后统计）。
- **Goodhart 申报**：借"通用"之名过度设计（YAGNI 违反），加入无人使用的参数与模式。对策：与"实现特化"配对检查——任何接口能力若无现存调用方且无易变性文档支持，退回。
- **状态**：T

### APOSD_07 不同层，不同抽象

- **原始智慧**：相邻层若抽象雷同，说明有一层没在干活；透传方法（方法体只是转发同签名调用）是典型症状（第 7 章）。
- **编译**：L3 —— 透传方法**可半机械检测**（方法体仅含一次转发调用、签名近似），是判断层红旗中少数能下推到静态门禁的，优先实现；装饰器/包装类数量异常同报。
- **验证信号**：透传方法密度。
- **Goodhart 申报**：在转发前后加一行无意义日志或空校验来骗过"纯转发"检测。对策：检测规则容忍装饰性语句；批判者复核被标记方法"这一层增加了什么抽象价值"。
- **状态**：T（静态检测部分可直接排期实现）

### APOSD_08 把复杂度往下拉

- **原始智慧**：模块的实现者多受苦一点，好过把苦难通过配置参数、异常、前置条件推给每一个调用方——因为调用方有千百个，实现者只有一个（第 8 章）。
- **症状**：配置参数增生；文档里大量"调用前必须……"；抛给调用方的异常种类繁多。
- **编译**：
  - L3 门禁：公开配置项数量棘轮（新增配置项需要在设计文档中论证"为什么这个决策不能在实现内做出"）；
  - L6 批判者："这个参数/异常/前置条件，是把哪个决策的成本从实现者转嫁给了调用方？实现内消化它的代价是什么？"
- **验证信号**：公开配置项与公开异常类型的数量趋势；调用方样板代码量。
- **Goodhart 申报**：把配置藏进环境变量、全局单例或"约定"来绕过配置项计数。对策：Semgrep 扫描环境变量与全局读取点，计入同一预算。
- **状态**：T

### APOSD_09 合并还是拆分

- **原始智慧**：拆分的默认收益是幻觉——拆分本身产生新复杂度（更多接口、更多依赖、逻辑分离）。判据是知识关联：理解 A 是否必须同时理解 B？共享信息多、总被一起用、一起才说得清的代码应该在一起（第 9 章）。conjoined methods 红旗：必须看 B 的实现才能懂 A。
- **编译**：L6 批判者——针对每次拆分/合并决策必答："这两块代码共享哪些知识？读者会不会在两个文件间来回跳转？"配合 git 度量：**总是被同一 commit 触碰的两个文件**是"该合未合"的强信号（可下推 L3）。
- **验证信号**：共同变更文件对（co-change pairs）的数量趋势。
- **Goodhart 申报**：为满足"低共同变更"而把本应同时改的代码改漏（更糟的结果）。对策：共同变更信号只用于**提示合并**，不用于惩罚同时修改。
- **状态**：T

### APOSD_10 把错误定义出局

- **原始智慧**：最好的异常处理是让异常不存在——重新定义语义使边界情况成为正常情况（unset 不存在的变量不该报错；substring 越界返回可用结果）。其次是异常聚合与就地屏蔽；最少化必须由调用方处理的异常（第 10 章）。
- **编译**：L5 流程 —— [设计文档五问](./README.md#设计文档五问流程类条目的共享工件)第 5 问："枚举这个接口的错误场景；对每一个，先回答**能否通过重新定义语义使它不再是错误**，不能再设计上抛。"
- **验证信号**：公开 API 上必须由调用方捕获的异常/错误码种类数。
- **Goodhart 申报**：把错误吞掉冒充"定义出局"（静默失败——比抛出更糟）。对策：这恰是 v1 规则 IMP_002 的门禁领地（空 catch、吞异常检测），L3 已有工具覆盖；"定义出局"必须改变**语义文档**而不只是删除 throw。
- **状态**：T

### APOSD_11 设计两次

- **原始智慧**：好设计不是第一个可行方案，而是备选比较后的幸存者；为重要抽象至少构思两个截然不同的设计再择优（第 11 章）。
- **编译**：L5 流程——计划阶段强制产出 2–3 个**结构上相异**的备选及权衡，单方案不予通过；否决理由写入[设计文档五问](./README.md#设计文档五问流程类条目的共享工件)第 3 问，沉淀为仓库记忆。
- **验证信号**：落选方案的事后翻案率（当初否决的方案后来被证明更好的频次——翻案率高说明当初的比较是走过场）。
- **Goodhart 申报**：稻草人备选——生成两个明显更差的陪跑方案衬托本命方案，比较即走过场。对策：批判者审备选的**相异性与竞争力**（备选若共享同一核心结构、或缺陷显而易见，不计为备选）；抽查时要求为落选方案作一轮辩护陈述。
- **备注**：本条目为对抗性评审发现 1 的修复——此前被 refactoring.md 以条目 ID 引用却并不存在，正是登记册自己定义的幽灵工件违规。
- **状态**：T

## 第三部分：注释、命名与显性（第 12–15、18 章）

### APOSD_12 注释先行是免费的设计验证

- **原始智慧**：注释的四个借口（"好代码自文档""没时间""会过时""没人读"）全部不成立；注释应当描述代码**说不出**的东西（意图、不变量、单位、边界语义）；接口注释先于实现写，因为**写不清注释 = 抽象没想清**——"hard to describe" 是设计红旗，不是文档问题（第 12、13、15 章）。
- **Agent 时代的关键转译**：对人，注释先行是需要自律的纪律；**对 Agent，它是免费的**——Agent 本来就先输出意图（对话、计划），强制它把意图写进接口注释，等于把意图从对话（回合结束即蒸发）迁移到工件（永久、可被下个会话读取）。这是"记忆外化"最便宜的实现。
- **编译**：
  - L5 流程：接口注释（含参数语义、单位、边界行为、不变量）先于实现生成，作为设计文档"调用方代码长什么样"一问的自然延伸；
  - L3 门禁：公开 API 缺文档注释即报错（现成 lint）；
  - L6 批判者：注释重复代码红旗（"注释只是把代码翻译成中文/英文"）；实现层细节污染接口注释红旗；注释与实现漂移检测（结合 git：代码改了注释没改）。
- **采集协议**：最小 L3 门禁已接入 `hooks/post_tool_use_quality_gate.py` 的 `MNT_002`，当前仅检查 Python 中通过 `__all__` 显式导出的函数/类是否缺 docstring；接口注释先于实现、注释漂移和盲测 oracle 仍待上层 harness 与抽查机制接线。
- **验证信号**：盲测 oracle 待建：从公开 API 注释生成调用任务，新会话 Agent 仅凭注释（不读实现）写调用代码，由测试或人工判卷确认是否正确。过渡期采用人工抽查，不把该信号当作自动门禁。
- **Goodhart 申报**：生成充满形容词的八股注释满足"有注释"门禁。对策：上述盲测协议抽查——八股注释无法让盲测 Agent 正确调用；失败的注释退回。oracle 未接线前只能作为审查任务，不得宣称已自动验证。
- **状态**：T

### APOSD_14 命名：模糊名是设计信号

- **原始智慧**：名字是最短的注释；起不出精确名字，通常说明这个东西职责不清——"hard to pick name" 与 "hard to describe" 同为设计红旗（第 14 章）。
- **编译**：
  - L3：模糊名清单（foo/bar/baz/temp/data/info/util/manager/helper/handler/process/flag…）与名字中的 `and` / 多动词模式静态可扫（v1 规则 IMP_011 扩表后保留）；
  - L6 批判者："这个名字在本仓库的统一语言（领域术语表）里吗？把名字遮住读方法体，你会起同一个名吗？"
  - 升级动作：批判者报"起名困难"时，**建议的不是换名而是拆分**——把设计信号用在设计上。
- **验证信号**：术语表覆盖率；模糊名密度。
- **Goodhart 申报**：机械加长名字（`processUserDataAndValidate`）冒充精确。对策：名字里出现 and/或多个动词本身入红旗清单。
- **状态**：T

### APOSD_18 代码应当显性

- **原始智慧**：不显性的代码要求读者补齐不在场的信息（第 18 章）。
- **编译**：语义转译 —— 显性 = 上下文足迹低，与 APOSD_02b 的晦涩度量合并，不单设机制。
- **状态**：I（并入 02b）

## 第四部分：修改、一致性与取舍（第 16、17、20、21 章）

### APOSD_16 修改现有代码时留在设计模式里

- **原始智慧**：系统的最终结构是无数次"顺手改一下"的累积；每次修改都要问"这是从零设计会得到的结构吗"，做不到就至少不劣化（童子军规则）；注释随代码走，别让文档漂移（第 16 章）。
- **编译**：L4 棘轮（触碰即不劣化，见 APOSD_03）+ L6 批判者对大 diff 追问："如果从零设计，这块会长这样吗？差距是什么？本次不弥合的理由？"——答案写进悔棋记录，成为仓库记忆。
- **Goodhart 申报**：以"本次不弥合"为万能豁免。对策：豁免必须附带 ADR 条目，同一处豁免第二次出现时升级为强制重构任务。
- **状态**：T

### APOSD_17 一致性是采样器配置

- **原始智慧**：一致性降低认知负荷——读者学一次规则，处处适用；不一致哪怕"更好"也常是负资产（第 17 章）。
- **编译**：L3（formatter/linter 管机械层）+ L7 范例（仓库既有模式即默认答案）+ L6 批判者："此 diff 与仓库现存的哪个既有模式偏离？偏离的理由是什么？"——**偏离合法，但必须付说明成本**。对 Agent 的额外意义见姊妹篇 3.3：一致的仓库让"对的那个工程师"持续出场。
- **Goodhart 申报**：为一致而复制现存的**坏**模式（一致性放大破窗）。对策：范例策展先行——坏模式要么修复要么在项目记忆中显式标记"勿模仿"。
- **状态**：T

### APOSD_20 性能：先测量，扔掉直觉

### APOSD_21 决定什么重要

- **原始智慧**：设计的核心能力是分辨重要与不重要——让重要的东西成为抽象的中心，把不重要的藏起来（第 21 章）。
- **编译**：**残渣为主**（README 第 4 节第 1、2 类）——"什么重要"的判断在人。可编译的一半：判断一旦做出，就写成工件（概念文档、优先级 ADR），此后批判者可以执行"这个改动把不重要的东西放到中心了吗"。
- **状态**：T
