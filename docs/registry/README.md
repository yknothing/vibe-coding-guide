# 编译登记册（Compilation Registry）

> 本目录是 [COMPILING-THE-CLASSICS.md](../COMPILING-THE-CLASSICS.md) 的活文档实现：把经典软件工程著作**逐书逐章**编译为环境构造，每个条目都可以直接被领走实现为门禁、hook、流程步骤或批判者提问。
> 理论（为什么编译）在姊妹篇里；这里只放材料（编译什么、编译成什么、怎么验证、会被怎么应付）。

---

## 1. 编译阶梯：往上推，能推多高推多高

六种编译形态（门禁/流程/提问/范例/经济学重估/语义转译）不是平级的。按**执行强度和成本**排序，存在一个阶梯：

| 层级 | 载体 | 执行强度 | 反馈延迟 | 例子 |
|---|---|---|---|---|
| L1 语言特性 | 语法/语义本身 | 违反需显式逃逸，逃逸可计数 | 即时 | Rust 所有权、Java records |
| L2 类型系统 | 编译期 | 编译失败 | 秒级 | TypeScript strict、NullAway |
| L3 静态门禁 | linter/静态分析/git 历史分析 | 报错回注 | 秒级 | Error Prone、clang-tidy、复杂度阈值、共同变更分析 |
| L4 动态门禁 | 测试/表征测试 | 红灯 | 分钟级 | TDD、表征测试、变更压测 |
| L5 流程步骤 | harness 硬步骤 | 不可跳过 | 回合级 | 强制备选、变绿后重构 |
| L6 批判者提问 | 干净上下文 LLM | 概率性 | 回合级 | 红旗清单、命名审查 |
| L7 范例与词汇 | 仓库策展 | 引导性 | 持续 | 正反例、统一语言 |
| L8 人 | 抽查/设计评审 | 权威但稀缺 | 天级 | 品味抽查、易变性输入 |

**上推原则**：每条智慧都应被推到它能到达的最高层级。能进语言的不留给 linter，能进 linter 的不留给批判者，能让批判者问的不占用人。层级越高，执行越确定、成本越低、越不消耗注意力。

两个推论：

1. **技术栈选择 = 预编译份额选择。** Rust 约等于"编译成语言的 Effective C++"——三/五法则、所有权纪律由借用检查器强制执行；TypeScript strict 模式编译了半本 JavaScript 陷阱清单；Java records 直接吸收了《Effective Java》的 equals/hashCode/toString 三个机械条目。**为 Agent 选择"无聊而严格"的技术栈，本质是采购一大块已经编译好的经典**——这从机制上解释了一线实践者"Agent 在无聊技术栈上表现更好"的观察。
2. **可编译边界在持续上移。** 一本书中只有可机械化的部分会消散于语言和工具；机械含量越高，"死"得越快。判断层密度高的书（如 APoSD）反而最长寿，这正是登记册把它列为头号工作面的原因。LLM 批判者（L6）是边界当前所在的位置——判断层知识第一次有了执行载体。

### ESC_01 逃逸舱口计数门禁（跨语言基础条目）

- **原始智慧**：语言与类型系统的强约束并非绝对墙；成熟语言都会保留逃逸舱口，例如 Rust `unsafe`、TypeScript `@ts-ignore` / `as any`、C/C++ `NOLINT`、Java `@SuppressWarnings`。这些逃逸是必要工程接口，但必须显式、可数、可审。
- **编译**：L3 门禁 —— 按语言扫描逃逸舱口数量与新增 diff，默认采用棘轮：新增逃逸必须带行内理由、issue/ADR id、作用域说明和过期条件；无理由逃逸阻塞，已登记逃逸进入周期审计。
- **验证信号**：逃逸舱口新增数、存量数、过期未清理数；逃逸行附近缺陷与返工密度。
- **Goodhart 申报**：把逃逸藏进封装函数、全局 helper 或生成代码目录，或者用空泛理由批量放行。对策：封装逃逸的 helper 也计入同一预算；生成目录需声明来源；理由模板必须引用具体风险与回收条件。
- **状态**：T

## 2. 条目 Schema

登记册复用本仓库 v1 规则集（`for-ai/rules/code_review_rules.md`）的分类学骨架——**v1 的 schema 指错了执行层，但 schema 本身是对的**，这里重新指向正确的层：

每个条目包含：

- **ID**：稳定标识（如 `APOSD_04`），供抑制注释、追踪、交叉引用使用；
- **原始智慧**：一句话陈述，注明出处章节；
- **症状**：违反时的可辨认信号（否定形知识远比肯定形可编译——优先记录红旗而非原则）；
- **编译**：目标层级（L1–L8）+ 形态 + 落地机制描述；
- **验证信号**：这条编译生效与否，用什么观测；
- **Goodhart 申报**（强制字段）：这条门禁/流程会被 Agent 怎么应付，以及对策。**没有申报 Goodhart 风险的条目不得进入 P 状态**——每个裁判都会被讨好，设计裁判时必须同时设计反讨好；
- **状态**：沿用 v1 的 `P`（生产可用）/`T`（试验中）/`E`（实验性）/`D`（已弃用），另增 `I`（索引/裁决记录——机制居于他处或仅记录一次取舍，豁免 Goodhart 申报与验证信号字段）。`P` 的授予条件是**机制已接线且第 3 节度量出现可归因变化**——仪表盘建成之前，任何条目都没有资格是 P；
- **采集协议**：T 条目的验证信号必须写明由谁、在什么事件、用什么命令采集；写不出采集协议的验证信号视为待建，条目不得晋升。

### 设计文档五问（流程类条目的共享工件）

多个流程条目共享同一份设计文档协议，唯一权威版本在此（引用请链接本节，勿复述）：

1. 这个模块**隐藏**了什么决策？一句话答不出 = 什么都没隐藏（APOSD_05）；
2. 哪些需求**易变**？由人注入——见第 4 节残渣第 2 类（APOSD_05）；
3. **否决**了哪些备选、为什么？（APOSD_11）
4. **调用方**的代码长什么样？接口与用例先于实现（APOSD_12）；
5. 枚举错误场景，每个先问：**能否通过重新定义语义使它不再是错误**？（APOSD_10）

## 3. 复杂度仪表盘：三症状 → 三类信号

Ousterhout 给复杂度下的操作性定义是三个症状。登记册把它们转成三类可采集信号，但不再声称三者都能从现有数据直接计算：有些信号只能做周期趋势，有些只能作为代理量，任何单一指标都不得独立裁决设计质量。

| 症状（APoSD 第 2 章） | 信号 | 采集协议 | Goodhart 申报 |
|---|---|---|---|
| 变更放大（改一个需求要碰很多处） | **变更放大系数**：每个逻辑变更平均触碰的文件/模块/API 面 | 由 release/PR harness 按 issue、spec 或 ADR id 聚合 commit；排除纯格式化和生成物；每个周期汇总一次 | 切细 issue、拆 commit、把同一逻辑变更伪装成多次小变更会刷低得分。对策：按逻辑变更 id 聚合，不按单 commit 计分。 |
| 认知负荷（理解一处代码要知道很多事） | **上下文足迹**：完成任务时实际加载的文件数、字节/token 量、工具读取记录、引用 hash | 由 agent harness 记录每回合读取文件、搜索命中、外部引用和上下文窗口使用量；只能和返工率、缺陷率、批判者发现联合判读 | 少读多猜、隐藏依赖、复用旧上下文都能让足迹变小但质量变差。对策：上下文足迹不得单独作为质量指标，必须和逃逸缺陷、返工、review 发现密度一起看。 |
| 未知的未知（不知道改动会波及哪里） | **缺陷逃逸相关性**：逃逸缺陷位置与近期变更位置、依赖边、共同变更文件对的距离 | 由缺陷回溯、`git blame`、测试失败归因和 issue 关闭记录周期性采集；每个缺陷必须记录发现来源和触发路径 | 饿死缺陷数据源、延迟登记、把改漏归因给环境都会让相关性好看。对策：缺陷来源清单固定，生产/用户/CI/人工抽查来源分开统计，缺失来源按 unknown 计入。 |

**职责拆分**：回合级棘轮只使用回合内可计算量，例如认知复杂度、函数长度、坏味道计数、修改行覆盖率、测试/变异得分。变更放大、上下文足迹、缺陷逃逸相关性是周期趋势、设计压测和校准信号，不直接阻塞单个回合。

当前最小采集原型：`tools/complexity_dashboard.py`。它生成 `complexity-dashboard/v1`
JSON 报告，并显式区分缺失数据、unknown 缺陷来源与已观测信号；这只是 APOSD_02a
的接线起点，条目晋升 `P` 仍需要周期趋势出现可归因变化。

## 4. 残渣：不可编译的部分

诚实的编译方案必须声明自己的边界。以下四类知识**原理上**无法编译进环境，它们定义了人的不可替代职责：

1. **问题构造**：决定"做什么"。规格可以驱动 Agent，但规格来自世界，不来自仓库。Brooks 在《没有银弹》里说的本质困难——概念构造——一大半在这里。
2. **易变性知识**：哪些需求会变、哪些决策要为未来留门。这是产品与市场知识，只能由人在设计时**注入**（设计文档的"什么易变"一问就是注入接口），无法由环境推断。
3. **概念完整性的所有权**。Brooks：好的设计出自一个（或极少数）头脑。Agent 舰队的民主式增量提交会稀释概念完整性——机制上只能缓解：单一规格谱系（所有特性必须能溯源到概念文档）、单一守护人、批判者检查"这个改动与系统的核心概念一致吗"。但**所有权本身不可下放给环境**。
4. **问责**。签名、发布、对后果负责——可以让 Agent 做一切，唯独不能让它"负责"。Willison 法则（不提交自己无法解释的代码）是问责在个人层面的形态。

另有一类**暂时残渣**：新领域没有任何验证信号时，第一个裁判必须由人建造（第一套测试、第一个度量、第一份红旗清单）。裁判建成后即可下放，但"建造第一个裁判"的能力始终在人。

## 5. 反身性：harness 自身也是软件

编译产物（规则、门禁、批判者、流程）本身会腐化，六条原理对它们同样适用：

- **规则数量棘轮**：常驻上下文的规则条数只减不增；新规则入库必须先回答"能否上推到 L1–L5"，并挤掉一条旧的（对应 AP1）。
- **批判者信用追踪**：低信用的批判者是"狼来了"——它的告警会被回路学会忽略，并连累其他告警的可信度。采纳裁定权不得交给被评生成回合：默认由人抽样裁定，或由不带生成上下文的第三上下文仲裁。每个批判者按**去重问题单元**统计采纳率，并与注入式召回测试配对：定期在评审样本里埋设已知缺陷，漏报计入召回失败。默认阈值为 `cfg: CRITIC_MIN_ADOPTION=0.5`，低于阈值且召回失败的批判者降级为 E 或废弃；只看精确率、不测召回会选育橡皮图章。
- **Gall 定律适用于 harness**：从最小可工作的门禁集演化，禁止一次性上线庞大的规则系统（本仓库 v1 的 90 条规则单体正是反例标本）。
- **条目生命周期**：语言/工具吸收了某条目（上推成功）就标记 D 并删除其下层实现。只有能指认吸收方、且下层规则或流程确实退役的上推式删除，才计入健康；合并、索引化、把条目搬到别处都不算"变短"。
- **文档语料受同一约束**：本仓库的文档不得引用不存在的工件（v1 README 之罪）；新文档对已有内容只许交叉引用不许复述；阈值类数字一律是可配置的默认值而非真理；合并前必须经过**干净上下文的对抗性评审**（生成者不得自审），评审记录入库于 `docs/reviews/`——不落成工件的评审等于没有发生。
