# 编译《Out of the Tar Pit》（Moseley / Marks）

> 选它的原因：这篇论文把复杂度的主要来源指向 state 与 control。Agent 时代代码生成更便宜，最容易无意识堆出更多可变状态、隐式控制流和派生数据。本登记册把它编译成状态预算与控制预算。
> 核查点：论文区分 essential / accidental complexity，强调 complexity 是大型软件成功开发的主要困难，并分析 state、control、code volume、data abstraction 等来源。

---

## TARP_01 State Budget：可变状态必须登记、归属、缩小

- **原始智慧**：state 是复杂度的核心来源之一；理解系统必须理解可能状态组合（论文关于 causes of complexity 的讨论）。
- **编译**：L3/L5 —— 新增可变状态必须声明 owner、生命周期、并发访问、持久化/恢复语义。默认优先不可变值、局部状态和单一写入者；共享 mutable state 进入预算。
- **验证信号**：共享可变状态点数量；无 owner 状态数量；状态相关缺陷数。
- **Goodhart 申报**：把状态藏进缓存、单例、闭包或外部服务。对策：缓存、全局对象、环境变量、数据库字段都纳入状态清单；"派生可重建"与"权威状态"分开标注。
- **状态**：T

## TARP_02 Derived State：能计算的不要存两份

- **原始智慧**：重复状态会制造一致性问题；数据一旦分叉，就需要控制逻辑维持同步。
- **编译**：L6 + L3 —— 批判者检查新增字段/缓存/索引是否可由已有权威数据计算；L3 标记同一概念在多个存储位置出现。
- **验证信号**：派生状态字段数；同步修复缺陷数；缓存失效事故数。
- **Goodhart 申报**：以性能为由持久化所有派生数据。对策：性能例外必须附 profile 证据、失效策略和重建路径。
- **状态**：T

## TARP_03 Control Budget：隐式控制流要显性化

- **原始智慧**：control 也是复杂度来源；隐式调用顺序、回调链、状态机分散都会提高理解成本。
- **编译**：L3/L6 —— L3 度量回调深度、事件链长度、状态机分散度；L6 问："读者从哪里看到完整控制流？失败路径是否与成功路径同样显性？"
- **验证信号**：跨文件事件链长度；隐式回调导致的缺陷数；状态机图是否存在并随代码更新。
- **Goodhart 申报**：把控制流藏进配置、注解或框架 magic，代码看似简单但行为不可见。对策：框架驱动路径必须有可生成/可维护的 route/job/event map。
- **状态**：T

## TARP_04 Declarative Core：把 what 从 how 里分离出来

- **原始智慧**：论文偏向用更声明式的方式表达系统，减少手写控制和状态管理；关系模型/函数式关系编程是其方案方向。
- **编译**：L5/L6 —— 对规则密集、查询密集、策略密集模块，先尝试表格/schema/rule DSL 表达 what，再生成或解释执行 how。批判者审："这段 imperative 代码是否其实是在维护一张规则表？"
- **验证信号**：策略变更是否只改数据/规则；规则覆盖测试生成率；手写分支数量下降。
- **Goodhart 申报**：为声明式而发明不可调试 DSL。对策：DSL 必须可 lint、可测试、可解释错误；小规模规则保留普通代码。
- **状态**：E

## TARP_05 Code Volume：少代码不是目标，少必须理解的状态/控制才是目标

- **原始智慧**：代码量会放大复杂度，但真正问题是理解负担；纯粹压缩代码可能更晦涩。
- **编译**：L6 —— 对"减少 LOC"的改动追问：少掉的是样板、重复知识、状态、控制，还是只是把逻辑压缩成更难懂的表达？
- **验证信号**：上下文足迹、认知复杂度、状态点、控制链长度是否下降；review 理解时间。
- **Goodhart 申报**：用 clever one-liner 或元编程减少 LOC。对策：LOC 不作为单独健康指标；必须配合可读性、状态/控制预算。
- **状态**：T
