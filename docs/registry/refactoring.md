# 编译《重构》（Refactoring, Fowler）

> 选它的原因：登记册里杠杆最大的机制——"变绿后强制重构"（[APOSD_03](./aposd.md#aposd_03-战略编程-vs-战术编程)）——的理论母体就是这本书。此外，第 3 章的坏味道目录（与 Kent Beck 合著）是"红旗式否定形知识"的祖师爷，是批判者清单最成熟的现成来源。
> 条目格式见 [README.md](./README.md) 第 2 节。

## FWL_01 两顶帽子：行为变更与结构变更不得混合

- **原始智慧**：任何时刻只戴一顶帽子——加功能（行为变化，测试跟着变）或重构（行为不变，测试不许变）。混戴是缺陷与不可审查 diff 的主要来源（第 2 章）。
- **编译**：
  - L5 流程：行为变更与结构变更分离为不同的 commit（Agent 回合内分阶段执行，产出分开）；
  - L3 代理门禁：由 **harness 按 diff 形状分类**，不采信 Agent 自贴的"重构"标签。纯结构变更不得触碰测试目录、测试期望、fixture、snapshot/golden、mock、测试辅助物、容差常量或行为配置；一旦触碰这些面，就不能按纯重构处理。
- **验证信号**：harness 分类出的纯结构 commit 比率；混合 commit 密度；触碰测试辅助面的重构候选数。
- **Goodhart 申报**：标签逃逸——把行为变更藏进未标 refactor 的 commit，或把测试辅助物改成"适配重构"来绕过"测试期望未动"。对策：分类不读标签，只读 diff shape、路径和语义规则；触碰无覆盖代码时 [FTH_01 表征测试](./legacy-code.md#2-fth_01-表征测试先行核心条目)先行激活，行为快照补齐后此门禁才有效——两个条目是依赖关系，不是并列关系。
- **状态**：T

## FWL_02 小步与持续绿灯

- **原始智慧**：重构是**一系列微小的、行为保持的变换**，每一步之后系统都可编译、可测试。大步重构失败时无处回退（第 2 章）。
- **Agent 时代的转译**：Agent 没有对大改动的恐惧感，天然倾向大爆炸重写；小步的真正价值在于**验证信号连续**——每步绿灯把"这次重构是否正确"从一个不可判定的大命题，分解为 N 个机器可判定的小命题。这是验证不对称性对重构过程本身的应用。
- **编译**：L5 流程——重构任务的回合结构为变换序列，每个变换后运行测试；某步红灯的处置是**回退该步**，而不是在红灯之上继续修补（红灯之上的修补就是混戴帽子）。
- **验证信号**：红灯后的处置方式分布：回退该步、修正测试、还是在红灯上硬修；目标是硬修比例下降，而不是步数变多。
- **Goodhart 申报**：步子切得无意义地琐碎以示合规。对策：步数不是指标，不进任何度量——唯一被观测的是绿灯连续性，切碎无利可图。
- **状态**：T

## FWL_03 坏味道目录：按编译阶梯分拣

- **原始智慧**：第 3 章的坏味道目录不是规则而是**嗅觉训练**——一组"该考虑重构了"的可辨认信号。它本来就是否定形知识，天然适合编译；正确动作是按阶梯分拣，而不是整体塞给任何单一载体：

| 坏味道 | 层级 | 机制 |
|---|---|---|
| 过长函数、过大类、过长参数列 | L3 | lizard / linter 阈值（v1 已有对应规则，保留） |
| 重复代码 | L3 | jscpd / Semgrep 克隆检测（v1 FND_002） |
| 霰弹式修改（一个变更碰 N 处） | L3 | git 共同变更分析——就是[变更放大系数](./README.md#3-复杂度仪表盘三症状--三类信号)的一个方向 |
| 发散式变化（一处因 N 种原因变） | L3 | git 分析的另一方向：同一文件被语义无关的需求反复触碰 |
| 依恋情结（方法更关心别人家的数据） | L3 + L6 | L3 用 ATFD 类度量（PMD/SonarQube 已有近似信号）筛出候选；L6 裁决"它是否真的想搬家" |
| 数据泥团（总是结伴出现的参数群） | L3 + L6 | L3 用 AST 统计参数/字段结伴出现频次；L6 裁决它们是否是一个没被命名的概念——与 [APOSD_14](./aposd.md#aposd_14-命名模糊名是设计信号) 同理 |
| 基本类型偏执 | L6 | 批判者审领域概念是否裸奔为 string/int |
| 过长消息链 | L3 | 静态可查（v1 IMP_012 需以静态 AST/调用链分析为主，LLM 只做误报裁决），与 [APOSD_07 透传](./aposd.md#aposd_07-不同层不同抽象) 互为亲戚 |
| 夸夸其谈的通用性 | L6 | 批判者——与 [APOSD_06](./aposd.md#aposd_06-略微通用的接口更深) 制衡：接口略通用合法，无调用方的实现能力非法 |
| 注释作除臭剂 | L6 | 见下方说明 |

- **关于"注释作除臭剂"与 APoSD 注释观的表面冲突**：Fowler 警告"需要大段注释解释的代码应该重构到不需要"，Ousterhout 强调"注释不可省"。两者不矛盾，裁决规则是：**注释描述代码"说不出"的东西（意图、单位、不变量）→ 必需（APOSD_12）；注释解释代码"本可以自己说清"的东西（这段循环在干嘛）→ 除臭剂，重构信号（FWL_03）**。批判者按此规则分流，两个条目不会互相打架。
- **Goodhart 申报**：改名除味——重命名、挪文件让味道在词面上消失而结构未变。对策：批判者核对的是结构证据（引用关系、参数分布、共同变更），不是命名。
- **状态**：T（L3 部分可直接排期）

## FWL_04 何时不重构（索引条目）

- **原始智慧**：不再触碰的代码不值得重构；重写有时优于重构（第 2 章"何时不该重构"）。
- **编译**：已由两处承接——"检查后无需重构"是合法结论（APOSD_03 的空产出条款）；重写 vs 微创的决策规则见 [legacy-code.md 第 6 节](./legacy-code.md#6-一处需要反向修正的地方)。本条目仅作索引，不设新机制（登记册纪律：能交叉引用的绝不复述）。
- **状态**：I（索引）

## FWL_05 演化式设计与"设计两次"的和解

- **原始智慧**：Fowler 主张设计通过持续重构**涌现**，弱化前期设计；Ousterhout 主张前期比较备选（design it twice）。人类时代这是真实的路线之争，因为两头都昂贵，必须选边。
- **编译**：经济学重估——生成廉价后**不再需要选边**：前期比较备选的成本是一个回合（[APOSD_11](./aposd.md#aposd_11-设计两次)），事后持续修正的成本由棘轮和重构阶段摊薄（[APOSD_03](./aposd.md#aposd_03-战略编程-vs-战术编程)）。正确姿势是两个都做：备选比较管方向性错误，演化重构管累积性劣化——它们防的本来就是不同的病。
- **状态**：I（裁决记录——机制已分别存在）
